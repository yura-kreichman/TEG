# Модуль «Билеты»

## Назначение

Продажа билетов заказами: клиент называет набор («1 на Колесо, 2 на Лодочки, 3 на Карусель»), кассир формирует заказ, клиент получает короткий номер заказа и (опционально) бумажные билеты, затем предъявляет их у актива. Продажа (деньги) и использование (гашение) — РАЗНЕСЕНЫ во времени и между операторами: этим режим отличается от launches/stays, где событие одно. Не POS: контроль — разницей при сдаче итогов.

## Режим учёта

Пятое значение `accountingMode = "tickets"`, рядоположное `counters`/`launches`/`cash_only`/`stays` (открытый список). Общее правило действует: режим меняется только пока у зоны нет ни одной сдачи итогов. В сегменте выбора режима — пятый пункт «Билеты».

## Цены — на активах, не тарифы

У tickets-зоны карточка «Тарифы» НЕ показывается (`Tariff` не используется). Вместо неё у каждого актива — ВАРИАНТЫ ЦЕН (сущность `TicketVariant`: название + цена + порядок; минимум один; например «Взрослый 50 / Детский 30 / Инвалид 0»; цена 0 допустима), редактируются в кебабе актива (рядом с существующим полем выбора тарифа, которое для tickets-зон скрывается). Билет фиксирует снапшоты `variantNameSnapshot`/`priceSnapshot` — смена/удаление вариантов не трогает проданное (паттерн `Launch.priceSnapshot`). Актив без единого варианта не может попасть в заказ — тайл неактивен с причиной.

## Заказ

Строго внутри одной tickets-зоны. Черновик-корзина живёт ТОЛЬКО локально во вкладке «Продать» (состояние страницы; перезагрузка теряет черновик — осознанно); сервер узнаёт о заказе атомарно в момент оплаты — «создан+оплачен» одной транзакцией, серверных черновиков нет. Позиции заказа = отдельные билеты («3 на Карусель» = три билета, каждый гасится отдельно). Оплата — ЦЕЛИКОМ ОДНИМ СПОСОБОМ: наличные / безнал / баланс (значения `cash`/`mobile`/`abonement`; переиспользовать компонент выбора оплаты и поиск кошелька по телефону из Товаров/Абонементов; списание с кошелька — при продаже, атомарно, `AbonementTransaction` `type="spend"` с новым полем `ticketOrderId` по аналогии с `launchId`/`goodsSaleId`; недостаточный баланс — отказ с показом остатка; частичная оплата не поддерживается; тумблер `Tenant.goodsAllowBalancePayment` для билетов не действует — это тумблер Товаров, для билетов отдельный тумблер не заводим, баланс доступен всегда: билеты — основная услуга точки, а не сопутствующая). Редактирование после оплаты — никакого; только аннулирование владельцем.

## Номер заказа

Короткий, пул отдельный на зону. Механика `smallestFreeNumber` (`src/lib/game-room.ts`, advisory-lock — тот же паттерн, лок по `zoneId`): «занят» = заказ, у которого есть хотя бы один ЖИВОЙ билет (статус `active`). Номер освобождается при завершении жизни заказа: все билеты погашены ИЛИ истекли ИЛИ аннулированы (в любой комбинации терминальных статусов). При выключенном гашении и бессрочной политике номера фактически растут — честная цена, отдельной логики не городить.

## Гашение — настройка зоны

`Zone.ticketRedemptionEnabled`, default `true`.

ВКЛЮЧЕНО: билеты имеют статусы `active`/`redeemed`/`expired`/`voided`; вкладка «Заказы» — полный цикл (гашение поштучно: кто, когда — `redeemedAt`/`redeemedByOperatorId`).

ВЫКЛЮЧЕНО («по старинке»): система знает только продажи; статусы не назначаются (все проданные — `active` бессрочно); вкладка «Заказы» существует, но READ-ONLY — ввод номера показывает состав и суммы заказа (для споров «пробили/не пробили»), без статусов и кнопок; поле «Срок жизни» в настройках скрыто и к заказам не применяется.

Деньги от тумблера не зависят — переключение безопасно в любой момент.

## Срок жизни

Только при включённом гашении. Настройка зоны `ticketLifetimeDays` (int, `null` = бессрочно). При продаже — снапшот `expiresAt` на заказе (дата продажи + дни; конец дня). На билете печатается «Действителен до: {дата}» (при `null` — строки нет). Истёкший билет: гасить нельзя (кнопка показывает «истёк {дата}»), визуально гаснет; статус `expired` назначается лениво при чтении (сравнение с `expiresAt`), отдельного крона нет. Деньги не трогаются — выручка учтена при продаже; возврат за истёкший — ручная воля владельца через аннулирование.

## Деньги и сдача итогов

Правки в `docs/spec/01-counters.md`. Зонный паттерн: операций в журнале Деньги поштучно НЕТ. Расчётная выручка = ПРОДАННЫЕ заказы с момента предыдущей сдачи (сумма итогов заказов; погашение на деньги НЕ влияет — решает подарочные/отложенные билеты). Способ оплаты заказа — справочная разбивка (нал/безнал/баланс), показывается подсказкой у полей кассы, БЕЗ автоподстановки (принцип 04-game-room: контроль независимым пересчётом); безнал и баланс — вне наличной разницы, как везде. Касса при сдаче — ОДНОЙ ПАРОЙ ПОЛЕЙ на зону (не по активам: заказ мультиактивный, физически деньги по активам не разложить — осознанное расхождение со stays/launches). Разница = факт − расчёт. Шаг возвратов/тестовых не показывается — его роль у аннулирования.

ДОСТУП К СДАЧЕ: tickets-зона показывается в мастере сдачи итогов ТОЛЬКО операторам с тумблером «Продажа билетов» (и владельцу) — оператор с доступом к зоне без тумблера гасит билеты, но зону в мастере не видит (прецедент: блок Товаров под тумблером; серверная фильтрация в submission-context, не только UI).

Аннулированные билеты исключаются из расчётной выручки того окна, в котором были проданы, если сдача ещё не сделана; если сдача уже прошла — аннулирование корректирующей роли в прошлых сдачах не играет (они неизменны), возврат — текущее событие кассы: `MoneyOperation`-расход из кассы зоны при наличном возврате / `refund` на кошелёк при балансовой оплате.

## Аннулирование — только владелец, поштучно по билетам

Погашенный билет не аннулируется (услуга оказана); активный и истёкший — можно; при выключенном гашении — любой (погашенных не существует). «Аннулировать заказ» = все живые билеты разом (кнопка-удобство). Возврат по `priceSnapshot` билета; балансовая оплата — `AbonementTransaction` `type="refund"` на сумму билета (паттерн Товаров). `CorrectionLog` (`entityType` `"TicketOrder"`/`"Ticket"` — согласуй с существующими значениями).

## PWA оператора

Зона tickets → экран с двумя вкладками (вход — по паттерну game-room: пункт в баре/списке зон, если есть доступ хоть к одной tickets-зоне; одна зона — сразу экран, несколько — список).

- **Вкладка «Продать»** (только с тумблером оператора «Продажа билетов»; без тумблера вкладка не рендерится, эндпоинт продажи проверяет на сервере): сетка тайлов активов (фото/цвет — существующая карточка актива) → тап по активу открывает варианты (чипы «название — цена») → тап по варианту добавляет билет в корзину; корзина — нижняя панель: позиции со степперами (актив · вариант · количество × цена), итог; кнопка «Оплатить» → выбор способа (существующий компонент; abonement → поиск кошелька) → успех: крупный номер заказа на весь экран + кнопка «Распечатать билеты» (если печать доступна: `printAvailability`-паттерн) + «Новый заказ». Черновик локальный; старт нового заказа очищает корзину.
- **Вкладка «Заказы»** (все операторы с доступом к зоне; существует всегда): главный элемент — ввод номера заказа (крупный циферблат) → карточка заказа: позиции-билеты (актив, вариант, цена; при включённом гашении — статус и кнопка «Погасить» у `active`-билетов: подтверждение внутри тайла билета, паттерн «Точно?» из game-room; при выключенном — read-only состав и суммы). Личная настройка вкладки — фильтр по активу («я стою на Карусели»: в карточках заказов чужие билеты приглушены, свои — первыми). Лента последних заказов зоны — ниже поля ввода, вторична. Гашение проверяется на сервере (статус, срок, права).

Вёрстка по `docs/spec/03-design-system.md`, строки через `/lang`.

## Печать

Мини-билет на КАЖДУЮ позицию заказа: крупный номер заказа, название актива, вариант, цена, дата продажи, «Действителен до» (если есть срок), брендинг тенанта (логотип/имя/подвал — существующий `ReceiptBranding`). Новый тип документа в `src/lib/print/receipt-document.ts` (`PrintDocumentData` уже поддерживает нужные поля — при необходимости расширь минимально); печать всех билетов заказа — N ПОСЛЕДОВАТЕЛЬНЫХ вызовов печати, по одному документу на билет (решение пользователя 2026-07-22: печать одним вызовом с @page-разрывами отклонена — риск многостраничной печати на слабых Bluetooth ESC/POS принтерах, подтверждённый живым багом в тот же вечер). Включение — существующие три уровня: `Tenant.printingEnabled` + `PointDevice.hasPrinter` + `Zone.printReceiptEnabled` (поле ПЕРЕИСПОЛЬЗУЕТСЯ; в настройках tickets-зоны подпись тумблера — «Печать билетов»). Кнопка печати также доступна из карточки заказа во вкладке «Заказы» (допечатать потерянный).

## Кабинет владельца

1. **Настройка зоны**: режим «Билеты» в сегменте; тумблер «Гашение билетов»; поле «Срок жизни заказов, дней» (видно только при включённом гашении); подпись печати «Печать билетов».
2. **Настройка актива**: варианты цен в кебабе (список название+цена, добавить/удалить/редактировать, полная замена набора при сохранении — паттерн `TariffOption`).
3. **Экран заказов tickets-зоны**: поиск по номеру, список за период, карточка заказа с билетами и статусами, аннулирование поштучно и «весь заказ», `CorrectionLog`.
4. **Тумблер «Продажа билетов»** в настройках оператора, группа «Работа» (второй модульный тумблер после Товаров — тот же UI-паттерн).

## Отчёты — без новых табов

1. Выручка tickets-зон входит во все денежные отчёты существующим зонным агрегатом (расчётная выручка = проданные заказы окна; сеансы = проданные билеты, штук) — money-роуты и «Касса за день» получают её автоматически через общий калькулятор (`results-calc`: добавь `isTicketsZone` и ветку агрегата).
2. «Показания по дням»: карточка сдачи tickets-зоны в паттерне `stays` — агрегат (заказов N · билетов M · разбивка способов справочно · разница), раскрытие разреза по активам и вариантам внутри карточки.
3. При включённом гашении — две строки в той же карточке: «Погашено X из Y · истекло Z». Никаких новых экранов и табов.

## Telegram

Только агрегат в сводке сдачи итогов: «🎫 {зона}: заказов N · билетов M · касса X vs расчёт Y · разница Z». Поштучных уведомлений нет.

## Не делаем (осознанно)

Серверные черновики корзины, редактирование оплаченного заказа, частичная оплата, смешанные способы оплаты, QR-коды на билетах (номер заказа — единственный идентификатор; QR — возможное будущее), перенос билетов между заказами, офлайн, скидки.

# Модуль «Товары»

## Назначение

Продажа сопутствующих товаров (бахилы, носки, кофе, сувениры) и товаров-услуг («Комплект для мастер-класса по лепке») на точках. Не POS с чеками: учёт продаж и остатков + лёгкая сверка кассы Товаров (см. «Сверка кассы» ниже), в философии остальных режимов учёта. Торговля — равноправная часть выручки точки, не отдельный бизнес (см. «Отчётность и размещение»).

## Каталог

Общий на тенанта. Товар: название, фото (опционально; плейсхолдер accent-soft при отсутствии), цена (Decimal, одна на все точки, задаёт владелец; БЕЗ истории цен — продажа фиксирует `priceSnapshot`, паттерн `Launch.priceSnapshot`), категория (обязательна), порог низкого остатка (опционально), учитывать ли остаток (bool, default true — товары-услуги остаток могут не вести). Категории создаёт владелец (название, порядок). Позиций 100+. Удаления нет — soft-delete `deletedAt` (паттерн `Tariff`/`Abonement`): архивный не продаётся, история неприкосновенна.

## Остатки

По точкам, только для товаров с учётом остатка. Модель МЯГКАЯ: продажа уменьшает расчётный остаток; блокировки при нуле НЕТ — при остатке <= 0 предупреждение на тайле. Пополнение: владелец, «+N штук» на точку, без закупочных цен (закупки вне системы). РЕВИЗИЯ ОСТАТКОВ: ввод фактического остатка ПО КАТЕГОРИИ (владелец всегда; оператор — при включённом тумблере товаров); по сохранении — таблица расхождений (расчётный / фактический / разница) и выравнивание расчётного на фактический; транзакционно; история ревизий с расхождениями хранится. (Ревизия остатков — про штуки на складе; не путать со «Сверкой кассы» ниже — та про деньги.)

## Доступ

Тумблер «Товары» в настройках оператора, группа «Работа» — первый в проекте булев тумблер уровня «весь модуль» у оператора (не переиспользует `allZonesAccess`/`allowedZones`, это список зон, а не флаг модуля; ближайший по духу, но не по форме, прецедент). Действует на ТРЁХ уровнях: (1) раздел продаж в PWA виден только с тумблером; (2) действие «Сверить кассу» (см. ниже) доступно оператору только с тумблером — без тумблера оператор товары не продаёт и кассу не сверяет вообще, только владелец может закрыть расхождение за него; (3) серверные проверки на всех эндпоинтах продаж, ревизий остатков и сверки кассы — доступ без тумблера отклоняется API, не только скрывается в UI.

## Продажа (PWA, раздел «Товары»)

Экран: поиск по названию (фильтр на лету) + горизонтальная лента чипов-категорий (паттерн CSS-слайдера) + сетка тайлов (фото, название, цена; сортировка внутри категории — по продажам за 14 дней, кэшируемый агрегат, ручной сортировки нет). Тап по тайлу → bottom sheet продажи: количество (степпер, default 1), итоговая сумма, ВЫБОР СПОСОБА ОПЛАТЫ — наличные / безнал / баланс: тот же UI-паттерн и значения `cash`/`mobile`/`abonement`, что при стопе пуска «По факту» и покупке абонемента — переиспользовать существующий компонент выбора оплаты. При «баланс»: поиск кошелька по телефону (существующий компонент из флоу абонементов); недостаточный баланс — продажа блокируется с показом остатка кошелька; частичная оплата не поддерживается. Продажа фиксирует: товар, количество, `priceSnapshot`, способ оплаты, кошелёк (для `abonement`), точку, оператора, серверное время. ОТМЕНЫ ОПЕРАТОРОМ НЕТ ни для какого способа оплаты — аннулирует только владелец.

## Деньги

Касса «Товары» — не новый слот в `MoneyOperation`, а переиспользование уже существующего паттерна: `MoneyOperation.zoneId`/`pointId` — ровно одно из двух заполнено (CHECK-констрейнт), `pointId`-слот уже используется точно так же операциями «Рабочего времени» (аванс/премия) и «Абонементов» (`abonement_topup`/`abonement_topup_cashless`, пополнение кошелька — операция без привязки к зоне). Продажа товара пишет `MoneyOperation` с `pointId` (без `zoneId`), новые типы: `goods_revenue` (наличные — физически в кассе точки), `goods_revenue_cashless` (безнал — учётно, добавляется в `CASH_EXCLUDED_TYPES`, `src/lib/zone-balance.ts`), `goods_revenue_abonement` (баланс — тоже в `CASH_EXCLUDED_TYPES`, деньги уже пришли раньше при пополнении кошелька, тот же принцип, что у `revenue_abonement`). Продажа за баланс — атомарно `AbonementTransaction` `type="spend"` (новое поле `goodsSaleId`, по аналогии с `launchId`; сумма = итог продажи) + `MoneyOperation` исключаемого типа, по образцу `src/lib/abonement.ts`; списание в момент продажи, не отложено.

`getPointCashBalance` (`src/lib/zone-balance.ts`) уже суммирует zone- и point-level операции в один физический остаток точки — `goods_revenue` попадёт туда автоматически, без изменения формулы суммирования. Единственная обязательная правка функции: `goods_revenue` нужно завести в ту же ветку отсечки по дате последней инкассации, что уже есть у `abonement_topup` (иначе кассу «Товары» никогда не занулит ни одна инкассация — там же и лежит настоящая, не мнимая, доработка кассового кода этого модуля).

Инкассация и расходы из кассы Товары — существующие механики (инкассация не делает сверку expected/actual ни для одной кассы в проекте, включая зоны — это отдельная забота «Сверки кассы» ниже, не инкассации). Настройка «из какой кассы списывать расходы» получает точку в список (**требует проверки на ШАГЕ 3** — при чтении кода на этапе обсуждения явного экрана с таким выбором вне мастера сдачи итогов не нашли; возможно, спека 02-money.md описывает то, что ещё не реализовано отдельным экраном — уточнить перед тем, как опираться на это поведение). В сводке остатков наличных точки — строка «в т.ч. касса Товары». Никакой товарной аналитики в разделе Деньги: Деньги отвечают только «сколько наличных где лежит».

## Сверка кассы (не в мастере сдачи итогов зон)

Решение, принятое в обсуждении перед этим шагом: Товары продаются на уровне ТОЧКИ, а не зоны, и не имеют естественного места в зоно-центричном мастере «Сдача итогов» (`ResultsSubmission` → `zoneSubmissions[]`, `docs/spec/01-counters.md`). Встраивать не стали — вместо этого у Товаров своя лёгкая, полностью независимая сверка:

- Новая сущность на точку (условно `GoodsReconciliation`), без связи с `ResultsSubmission`. Курсор — «с момента предыдущей сверки этой точки» (та же идея, что «с момента предыдущей сдачи» у Пусков/Прибываний, только по точке в целом).
- Кто может провести: владелец — всегда; оператор — только с тумблером «Товары».
- Экран: расчётная выручка Товаров с прошлой сверки — READ-ONLY, разбита на нал/безнал/баланс. Баланс показывается справочно, в саму сверку не входит (деньги пришли раньше, при пополнении кошелька — тот же принцип, что уже действует у зон с абонементной оплатой). Нал и безнал вводятся вручную, БЕЗ автоподстановки из расчёта — самостоятельный ввод и есть контроль (тот же принцип, что у Прибываний/Пусков, `docs/spec/04-game-room.md`, «Только справочная разбивка, не автозаполнение»). Разница = (факт нал + факт безнал + расчётный баланс) − расчётная выручка.
- Когда: не привязана к сменам, доступна по кнопке в разделе «Товары» в любой момент — естественно тяготеет к моменту инкассации, но не требует её.
- Что пишет: сама сверка НЕ создаёт `MoneyOperation` — продажи уже создали свои операции в момент продажи, сверка только фиксирует факт проверки и расхождение. История хранится (тот же принцип, что у истории ревизий остатков выше).
- Telegram/email: короткое сообщение по факту сверки (не по факту сдачи итогов зон, т.к. с ней больше не связана) — «ТОВАРЫ: сверка · расчёт X · факт Y · разница D». Как и у остальных сводок в проекте, формат — дело кода (`src/lib/summary-channels/`), не отдельного файла спеки (у системы сводок его нет вообще, см. `docs/spec/01-counters.md`, раздел «Сводки»).

## Аннулирование (только владелец)

Список продаж: аннулирование с подтверждением и записью в `CorrectionLog` (`entityType: "GoodsSale"`); возвращает остаток (если товар с учётом остатка); для продажи за баланс — сторнирует списание (возврат суммы на кошелёк, тип транзакции `refund` со ссылкой на аннулированную продажу — консистентно с существующими `topup`/`spend`) и компенсирующую `MoneyOperation`.

## Кабинет владельца

Раздел «Товары» (в «Ещё»), вкладки:
- **Каталог** — CRUD товаров/категорий, архив.
- **Остатки** — по точкам, маркеры низкого остатка, пополнение, ревизия остатков (по категории), история ревизий с расхождениями.
- **Продажи** — шапка-сводка за выбранный период (продаж шт · выручка · нал/безнал/баланс) + расчётная касса с прошлой сверки и кнопка «Сверить кассу» (действие из раздела «Сверка кассы» выше — живёт здесь, не отдельной вкладкой, т.к. это про те же деньги, что и список продаж); под сводкой — список продаж с фильтрами точка/категория/товар/оператор/способ оплаты, аннулирование; история сверок с расхождениями.

## Отчётность и размещение

*Отдельного файла спеки Отчётов в проекте нет (отчётная логика описана кусками в 01-counters.md/02-money.md) — раздел ниже живёт здесь, а не как правка стороннего файла, до тех пор, пока такой файл не появится.*

Принцип: выручка точки ЕДИНАЯ — торговля не отделяется от проката.

1. «Выручка по дням» и все сравнения периодов: товары — равноправный слой стека рядом с зонами.
2. «Топ товаров»: горизонтальный рейтинг по выручке с переключателем на штуки, фильтры категория/точка — уровень 2.
3. «Способы оплаты товаров»: нал/безнал/баланс по периодам; доля баланса — индикатор работы программы абонементов — уровень 2.
4. «Расхождения ревизий остатков» — история расчёт vs факт по категориям и точкам — уровень 3 (контроль).
5. «Расхождения сверок кассы» — история расчёт vs факт по точкам, из «Сверки кассы» выше — тоже уровень 3 (контроль), отдельно от пункта 4: одно про штуки на складе, другое про деньги.

Иерархия размещения (общий принцип, годится не только для Товаров — кандидат на перенос в `00-architecture.md`, когда там появится подходящее место, либо в будущую спеку Отчётов): Деньги — где лежат наличные; раздел модуля — оперативная работа и сводка за период; Отчёты — тренды и сравнения.

## Не делаем (осознанно)

Отмена оператором, штрих-коды и сканирование, скидки/акции, модификаторы, поставщики/накладные, сроки годности, закупочные цены и маржа, per-точка цены, частичная оплата балансом, офлайн. Также не делаем: встраивание в мастер «Сдача итогов» зон (решение этого шага — см. «Сверка кассы» выше).

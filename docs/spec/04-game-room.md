# Модуль «Игровая комната»

## Назначение

Оперативный учёт сеансов в реальном времени: оператор фиксирует каждый пуск в PWA в момент события — старт и стоп, а не показание счётчика или ручной итог. Живой вариант режима `launches` (модуль «Счётчики», `docs/spec/01-counters.md`) — **не отдельный режим учёта зоны**, а его суб-вариант, наравне с существующим «ручным» вводом готового итога заездов. Никаких «рекомендованных» вариантов и подсказок о предпочтительности.

Типичный кейс — детская игровая комната: у каждого ребёнка на входе браслет/бирка с номером, оператор стартует пуск по бирке и стопает, когда ребёнок уходит (принимая деньги на месте, вне системы — см. «Тариф зоны» ниже).

## Режим учёта — вариант launchMode у accountingMode="launches"

Свойство зоны `accountingMode` (`docs/spec/01-counters.md`, «Режим учёта зоны») **не меняется** — Игровая комната не добавляет новое значение в этот открытый список. Вместо этого у зоны появляется новое поле `launchMode`, осмысленное только при `accountingMode = "launches"`:

- **`manual`** (по умолчанию) — существующее поведение `launches`: оператор вводит готовое число заездов за сдачу.
- **`game_room`** — учёт в реальном времени: активы (опционально) и один тариф на зону остаются для детализации; вместо ручного ввода итога — оператор стартует и стопает каждый пуск отдельно, система сама считает количество и сумму.

Оба варианта — это один и тот же `accountingMode = "launches"` с точки зрения 01-counters.md (цепочки показаний нет, возвраты/тестовые — своя логика per-variant, см. ниже); `launchMode` — деталь именно этого режима, не самостоятельная ось. В интерфейсе выбор двухуровневый: сначала «Счётчики / Пуски / Только касса», затем, если выбраны «Пуски» — «Вручную / Игровая комната».

Переключение `launchMode` (и `accountingMode` в целом) свободное, действует с момента переключения: прошлые сдачи итогов неизменны. Редактируется только пока у зоны нет ни одной сдачи итогов — то же правило, что и у `accountingMode` (см. 01-counters.md, «Режим учёта зоны»).

## Тариф зоны (один из двух, выбирает владелец)

- **ФИКСИРОВАННЫЙ (fixed)** — цена за пуск. Опционально длительность в минутах: тогда у активного пуска обратный отсчёт, по истечении — сигнал оператору (звук + вибрация + тайл краснеет и пульсирует). Пуск НЕ завершается автоматически по истечении — только оператором.
- **ПО ВРЕМЕНИ (per_minute)** — тариф за минуту, стоимость = длительность × тариф. Настройки: округление длительности до минуты (вверх / вниз / математически) и минимальная сумма пуска. Живой тайл показывает время и текущую сумму справочно.

В отличие от `counters`, где у зоны может быть до двух тарифов (два RFID-считывателя на актив), у `launchMode="game_room"` — **один** тариф на зону: осознанное упрощение MVP, fixed и per_minute взаимоисключающие, не два одновременно. Можно вернуться к паре тарифов после MVP при реальном запросе.

Момент оплаты система НЕ фиксирует (не POS, деньги принимаются оператором на месте вне системы) — считается только расчётная выручка, контроль — по разнице при сдаче итогов, как в `counters`/`launches`.

## Пуск

Привязка: к активу зоны, если активы есть; если у зоны нет активов — к зоне целиком (детская игровая комната обычно без активов — один общий тайл).

Один актив/зона может держать несколько параллельных открытых пусков (несколько детей одновременно): автонумерация в рамках актива/зоны (Пуск 1, 2, ...), присваивается **сервером атомарно** (два одновременных старта с разных устройств не должны получить один номер) + опциональная короткая метка — одно текстовое поле при старте, например номер браслета («браслет 12», «синяя куртка»).

Пауз нет: время течёт от старта до стопа. Источник времени — **сервер**, тот же паттерн, что открытые смены авто-режима (`docs/spec/05-work-time.md`): `startedAt`/`endedAt` ставятся сервером, клиентский таймер — только отображение от серверного `startedAt`, переживает перезапуск PWA и смену устройства.

## Жизненный цикл

Старт и стоп — оператор. Правка времени и аннулирование — ТОЛЬКО владелец, с журналом правок (кто, когда, было → стало; оператору — компактный tooltip, тот же паттерн, что у показаний `counters`, см. 01-counters.md «Прозрачность»).

Пуск длиннее порога зоны (настройка `longLaunchThresholdMinutes`, по умолчанию 60 мин) помечается: оператору — выделение тайла, владельцу — маркер «требует правки» в списках.

Автостопа нет — время не выдумывается.

## Деньги и сдача итогов

*Согласующие правки в `docs/spec/01-counters.md`, раздел «Сдача итогов» — см. отдельный список изменений ниже.*

Пуски не создают операций в журнале Деньги поштучно — как и показания `counters`. Для зоны с `launchMode = "game_room"`:

- Шаг 2 мастера сдачи итогов (показания/заезды) скрыт целиком — как у `cash_only`, но, в отличие от него, расчётная выручка и разница существуют.
- Расчётная выручка = агрегат завершённых пусков зоны с момента предыдущей сдачи:
  - **fixed**: количество завершённых пусков × цена;
  - **per_minute**: сумма стоимостей завершённых пусков (у каждого — уже применённые округление и минималка на момент закрытия).
- Шаг 3 мастера («возвраты/тестовые») для зон с `launchMode = "game_room"` не показывается — его роль выполняет аннулирование пуска владельцем (аннулированный пуск целиком исключается из агрегата). У `launchMode = "manual"` возвраты/тестовые остаются, без изменений.
- Шаг 4 (касса) и шаг 5 (расходы) — без изменений.
- Разница = фактическая касса (наличные + безнал) − расчётная выручка; вся логика контроля (кто видит, что делает владелец) — как в `counters`/`launches`.

**Мягкая блокировка**: сдача итогов по зоне недоступна при открытых пусках — сообщение «Заверши N активных пусков», без обхода.

В отчётах пуски и показания `counters`/`launches` агрегируются как единые «сеансы»; для зон с `launchMode = "game_room"` дополнительно доступно суммарное время (Xч Yм).

Сводка (Telegram/email) после сдачи итогов — для зон с `launchMode = "game_room"` вместо блока показаний: «Пусков: N · время: Xч Yм».

## Экран зоны в PWA

Сетка тайлов активов (фото + цветовая метка — как в `counters`) — если у зоны есть активы. Зона без активов (типичный кейс игровой комнаты) — один большой тайл зоны с той же механикой.

Тайл без активных пусков: тап → старт (per_minute и fixed без длительности — мгновенно; опциональная метка — вторым касанием в поле на тайле или в sheet быстрого старта; старт не должен требовать более двух тапов).

Тайл с активными пусками: количество и ближайший таймер; тап → раскрытие списка пусков (номер, метка, время, текущая сумма/отсчёт); у каждого — кнопка «Стоп» → подтверждающий bottom sheet: длительность и итоговая сумма (per_minute — с применённым округлением и минималкой), кнопка «Завершить».

Истёкший отсчёт (fixed с длительностью): звук + вибрация + пульсация тайла и строки пуска. Звук воспроизводится только при открытом PWA (Web Audio, разблокируется первым взаимодействием пользователя с приложением) — ограничение платформы, зафиксировано здесь и в подсказке оператору при первом использовании режима. `prefers-reduced-motion` — без пульсации, сигнал остаётся звук + вибрация.

Вёрстка по `docs/spec/03-design-system.md`, все строки — через `/lang`.

## Кабинет владельца

1. **Настройка зоны**: режим учёта (сегмент из трёх: «Счётчики»/«Пуски»/«Только касса»; при выборе «Пуски» — второй уровень «Вручную»/«Игровая комната»), тариф (fixed/per_minute), цена/тариф, длительность (для fixed), округление и минималка (для per_minute), порог долгого пуска.
2. **Экран «Сейчас на точке»**: живые активные пуски по зонам (актив/зона, номер, метка, время; обновление поллингом 10–15 сек, без реалтайм-выручки), плюс счётчик «пусков сегодня» по зонам. Вход — с Главной владельца, карточка видна только при наличии зон с `launchMode = "game_room"`.
3. **Список пусков за период** с фильтрами (зона, дата): правка времени, аннулирование (с подтверждением и журналом); аннулированные исключаются из расчётной выручки.

## Ограничения MVP (осознанные решения)

- Офлайна нет — без сети старт/стоп невозможны.
- Telegram/email-уведомлений по каждому отдельному пуску нет — только агрегат в сводке сдачи итогов.
- Паузы нет.
- Клиентского выбора тарифа при старте нет — тариф зоны один, задаётся владельцем заранее.
- Один тариф на зону (без пары «детский/взрослый», как у `counters`).

# Модуль «Счётчики»

## Структура

Точка → Зоны (аттракционы) → Активы (устройства).

У зоны 1 или 2 тарифа: название свободным текстом + цена за сеанс. При двух тарифах у каждого актива зоны — два программных счётчика (два RFID-ключа на устройстве). При достигнутом лимите (2 тарифа) кнопка добавления тарифа не скрывается, а показывается заблокированной с текстом «Достигнут лимит: 2 тарифа» (см. `docs/design/design-system.md`).

Карточка актива: название, опциональное фото, цветовая метка.

Счётчики накопительные, только вперёд, 4 разряда; переполнение 9999→0 учитывается в расчёте (разница считается по модулю 10000).

Владелец создаёт и редактирует зоны, активы, тарифы; может корректировать показания (например, после физического сброса счётчика PIN-кодом на устройстве).

## Сдача итогов (основная операция)

Пошаговый мастер в PWA оператора:

1. Выбор работавших зон из доступных оператору. Невыбранная зона = не работала, отдельного статуса нет. UI: плитки зон (см. `docs/design/design-system.md`, «Тёмная тема (PWA оператора)») — тап переключает выбор, отмеченная плитка получает акцентную заливку и галочку.
2. По каждой зоне — показания всех активов (1 или 2 числа на актив, по числу тарифов). При вводе показывать фото и цветовую метку актива. UI: плитки активов, тап открывает bottom sheet с полями по тарифам; под каждым полем — живой пересчёт числа заездов (с учётом перехода счётчика через 9999→0); заполненная плитка получает галочку.
3. Возвраты/тестовые пуски — одно числовое поле (шт.) на зону.
4. Касса зоны: наличные и мобильный платёж раздельно.
5. Расходы: сумма + необязательный комментарий, записей может быть несколько (сохраняются как операции модуля Деньги).

Сдач в день может быть несколько (пересменка). Расчёт всегда от предыдущей сдачи, независимо от даты.

## Расчёт

Сеансы актива = текущее показание − показание предыдущей сдачи (mod 10000).

Расчётная выручка зоны = Σ по тарифам: (сеансы − возвраты/тесты) × цена.

Разница = фактическая касса (наличные + мобильный) − расчётная выручка; может быть любого знака — система только показывает, решения по разнице принимает владелец.

## Валидация

Показание меньше предыдущего → предупреждение, отправка всё равно возможна.

## Настройки владельца

- Доступ оператора к зонам (все / выборочно).
- Показывать ли оператору предыдущие показания (в т.ч. чужой сдачи при пересменке).

## Прозрачность

Правки записей — только владелец. Журнал правок: кто, когда, было → стало.

Оператор видит факт правки (компактно, tooltip у значения), владелец — полную историю.

Правка/удаление сдачи доступны только владельцу и только для **последней** сдачи в цепочке показаний по её активам — если по хотя бы одному активу зоны уже есть более поздняя сдача, кнопки блокируются (нельзя задним числом менять базу, от которой считалась следующая сдача). Экран: `docs/design/prototype-owner-readings-v1.html` — «Показания по дням» (`/money/readings`), карточка на зону-сдачу с цепочкой «пред. → тек.» и дельтой по каждому тарифу, kebab открывает «Изменить показания и сумму» (предзаполненная форма, необязательная причина правки) или «Удалить сдачу» (с подтверждением); у заблокированной сдачи вместо действий — поясняющая заметка.

## Telegram

Сводка после каждой сдачи итогов — в канал/чат тенанта:
- блоки по зонам: показания активов, касса vs счётчики, разница, возвраты/тесты;
- общий блок «Касса»: имя оператора, наличные/мобильный/расходы раздельно.

Токен бота и chat_id — настройки тенанта; хранить шифрованно/в env-механизме, никогда не в коде.

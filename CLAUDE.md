# RentOS

SaaS для учёта детского проката (электромобили, батуты, квадроциклы, игровые комнаты, VR и т.д.) — мультитенантный, модульный. Домен: RentOs365.app.

## Стек

- Next.js 16 (App Router) + TypeScript + Tailwind CSS
- PostgreSQL 16 в Docker (`docker-compose.yml`) + Prisma 7 (генератор `prisma-client`, обязателен driver adapter — `@prisma/adapter-pg`; голый `new PrismaClient()` без адаптера не соберётся)
- Свой код авторизации, без сторонних библиотек (NextAuth/Clerk и т.п.) — `src/lib/auth.ts`

## Терминология

«**Сдача итогов**» (results submission) — кассовое событие модуля Счётчики.

Слово «**смена/shift**» зарезервировано за будущим модулем «Рабочее время» — в модуле Счётчики не используется никогда: ни в UI, ни в именах переменных/таблиц, ни в комментариях.

## Проверки модуля «Лендинг»

При любых будущих изменениях `/site/[slug]` (публичная страница, docs/spec/08-landing.md) — обязательно перед тем, как считать задачу законченной:
- Прогнать Lighthouse (mobile): Performance, SEO, Best Practices >= 90; LCP < 2.5s.
- Провалидировать JSON-LD выдачи страницы через валидатор структурированных данных.
- Проверить вкладку Network реального браузера — ноль запросов к внешним доменам (кликабельные ссылки на карты/соцсети — не в счёт, это не автоматические запросы).
- Реальный бюджет JS для этого стека (Next.js App Router) — не «< 100KB» из исходной формулировки, а **~200KB gzip** (замерено 2026-07-13 — это фреймворковый минимум React+ReactDOM+Next-рантайма, не код лендинга; собственный код добавляет ~0KB сверху, пока не заведена ни одна React "use client" граница на этом маршруте — см. `public/landing-share.js`, сознательно ванильный JS, не React-компонент).

Публичные страницы (лендинг, инструктажи) верстаются по 03-design-system.md наравне с кабинетом; при любом изменении публичных страниц — проверка соответствия и Lighthouse.

## Принципы

- **Модульность** — ядро (тенанты, пользователи/роли, точки, зоны, активы, настройки) + подключаемые модули; каждый модуль объявляет свои сущности, экраны и права ролей, включается на уровне тенанта feature-флагом; добавление модуля не должно требовать правок ядра.
- **Мультитенантность** — все данные скоуплены на тенанта. Вход не единый: у Owner/Super Admin — обычный аккаунт (email+пароль, PIN — личное удобство на своём устройстве); у Operator — отдельная модель без email, PIN уникален только внутри тенанта, назначается и меняется исключительно Owner'ом (до ~50 операторов на тенанта), вход — с устройства точки, привязанного к тенанту, а не к конкретному человеку. Подробности и статус реализации — в `00-architecture.md` и в плане аудита.
- **i18n** — все строки UI только через ключи из `/lang/*.json` (`ru.json` — базовый язык разработки, `en.json`, `ro.json`...). Пользовательские данные (названия зон/активов/тарифов, комментарии) не переводятся.
- **Темы** — светлая/тёмная через CSS-переменные (design tokens), модули наследуют тему автоматически.

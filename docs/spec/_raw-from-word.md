**АРХИТЕКТУРНЫЕ ПРИНЦИПЫ (сквозной раздел ТЗ)**

**Модульность.** Система строится как ядро + подключаемые модули. Ядро: тенанты, пользователи и роли, точки, зоны, активы, настройки. Модули: Счётчики, Деньги, Рабочее время, Сервис, QR-платежи, Задания, Игровая комната, Печать билетов (нефискальных), Отчёты — список открытый, добавление нового модуля не должно требовать изменения ядра. Каждый модуль объявляет: свои таблицы/сущности, свои экраны в PWA и кабинете владельца, свои права для ролей, свои записи в общие журналы (например, денежные операции). Модули включаются/выключаются на уровне тенанта флагами (feature flags).

**Подписки и лимиты.** Пакет подписки = набор включённых модулей + числовые лимиты: количество точек, зон, активов (счётчиков), операторов. Лимиты проверяются при создании сущностей («в вашем пакете до 15 счётчиков»). Статусы подписки: active, paused (сезонная пауза), expired. Конфигурация пакетов не зашита в код — управляется из админ-модуля.

**Админ-модуль (Super Admin).** Отдельный интерфейс владельца платформы: тенанты и их статусы, конфигуратор пакетов (модули + лимиты + цены), включение модулей конкретному тенанту вручную (индивидуальные условия), базовая статистика использования.

**Мультиязычность.** Все строки интерфейса — в языковых файлах JSON в папке /lang (ru.json, en.json, ro.json...). В коде только ключи. Язык — настройка тенанта с возможностью личного переопределения пользователем. Данные пользователей (названия зон, активов, тарифов, комментарии) не переводятся.

**Темы.** Светлый и тёмный режим во всех интерфейсах, переключение пользователем, реализация через CSS-переменные (design tokens), чтобы модули автоматически наследовали тему.

**МОДУЛЬ «СЧЁТЧИКИ»**

**Структура.** Точка → Зоны (аттракционы) → Активы (устройства). У зоны 1 или 2 тарифа: название свободным текстом + цена за сеанс. При двух тарифах у каждого актива зоны два программных счётчика (два RFID-ключа на устройстве). Карточка актива: название, опциональное фото, цветовая метка. Счётчики накопительные, только вперёд, 4 разряда; переполнение 9999→0 учитывается в расчёте (разница по модулю 10000). Владелец создаёт и редактирует зоны, активы, тарифы; может корректировать показания (например, после физического сброса счётчика PIN-кодом на устройстве).

**Сдача итогов — основная операция.** Оператор в PWA: выбирает, какие из доступных ему зон работали → по каждой зоне вводит показания всех активов (одно или два числа на актив — по числу тарифов) → количество возвратов/тестовых пусков (одно поле, шт.) → наличные и мобильный платёж раздельно → расходы (сумма + необязательный комментарий, записей может быть несколько; сохраняются в модуль «Деньги»). Пошаговый мастер; при вводе показаний отображается фото и цветовая метка актива. Сдач в день может быть несколько (пересменка); расчёт всегда относительно предыдущей сдачи, независимо от даты. Невыбранная зона = не работала, отдельный статус не нужен.

**Расчёт.** Сеансы актива = текущее показание − показание предыдущей сдачи (mod 10000). Расчётная выручка зоны = Σ по тарифам (сеансы − возвраты/тестовые) × цена. Разница = фактическая касса (нал + мобильный) − расчётная выручка. Разница может быть положительной и отрицательной; система только показывает её, решения — за владельцем.

**Валидация.** Показание меньше предыдущего → предупреждение, но отправка возможна.

**Доступы и настройки владельца.** Каким зонам оператор имеет доступ для сдачи итогов (все / выборочно). Показывать ли оператору предыдущие показания (в т.ч. чужой сдачи при пересменке) — настройка тенанта.

**Прозрачность.** Правки записей — только владелец. Каждая правка в журнале: кто, когда, было → стало. Оператор видит факт правки компактно (tooltip у значения), владелец — полную историю.

**Вход операторов.** Аккаунты создаёт владелец. PIN уникален в рамках тенанта (у разных тенантов PIN могут совпадать). Привязка к тенанту — через установочную ссылку/QR точки (PWA ставится один раз при найме) или через активированное владельцем «устройство точки». Деактивация PIN мгновенно закрывает доступ.

**Telegram-сводка.** Отправляется по факту каждой сдачи итогов в канал/чат тенанта: блоки по зонам (показания активов, касса vs счётчики, разница, возвраты/тесты) + общий блок «Касса» с именем оператора, наличные/мобильный/расходы раздельно.

**МОДУЛЬ «ДЕНЬГИ»**

**Назначение.** Единый журнал движения наличных и остатки касс. Другие модули не ведут свои деньги, а пишут операции сюда.

**Кассы.** У каждой зоны своя наличная касса; остаток по зоне и общий по точке считаются из журнала.

**Типы операций.** Выручка (создаётся автоматически при сдаче итогов: наличные — в кассу зоны, мобильный — учётно, без наличного остатка); Расход (вводится при сдаче итогов; из кассы какой зоны списывать — настройка владельца); Инкассация (оператор вводит переданную владельцу сумму, касса уменьшается; подтверждение владельцем не требуется — расхождения решаются на месте); Размен (владелец фиксирует внесение наличных на точку). Формат записи: дата-время, тип, сумма, касса/зона, кто провёл, комментарий. Список типов расширяемый — будущие модули добавляют свои (например, аванс сотруднику из кассы).

**Отчётность.** Журнал — источник для отчёта «Бизнес: расходы и прибыль» и для текущего остатка «сколько наличных должно быть на точке».

# Модуль «Прибывания»

## Назначение

Оперативный учёт сеансов в реальном времени: оператор фиксирует каждый пуск в PWA в момент события — старт и стоп, а не показание счётчика или ручной итог готового числа заездов. «Прибывания» (`accountingMode = "stays"`) — самостоятельный режим учёта зоны, рядоположный `counters`/`launches`/`cash_only` (решение пользователя 2026-07-17; ранее был суб-вариантом `launches` под именем «Игровая комната» — пересмотрено). Никаких «рекомендованных» режимов и подсказок о предпочтительности.

Типичный кейс — детская игровая комната: у каждого ребёнка на входе браслет/бирка с номером, оператор стартует пуск по бирке и стопает, когда ребёнок уходит (принимая деньги на месте, вне системы — см. «Тариф» ниже).

`accountingMode = "launches"` («Пуски», `docs/spec/01-counters.md`) переиспользует ту же модель `Launch` и тот же принцип агрегата с момента предыдущей сдачи (запрос пользователя 2026-07-17), но без концепции браслета и без старта/стопа во времени: тап по активу мгновенно закрывает пуск (`startedAt === endedAt`) по одному из до-двух ПЛОСКИХ тарифов зоны (`Tariff.pricingMode = null`, без вариантов длительности — тариф выбирается на каждом тапе, не привязан к активу заранее). PWA-экран — `/operator/launches`, тайлы активов с живым счётчиком пусков вместо браслетов. Формулы расчёта агрегата (см. «Жизненный цикл сеанса» ниже) — общие для `stays` и `launches`.

## Режим учёта — accountingMode="stays"

Свойство зоны `accountingMode` (`docs/spec/01-counters.md`, «Режим учёта зоны») — открытый список, рядоположные значения: `counters` | `launches` | `cash_only` | `stays` | `tickets` (Билеты, `docs/spec/10-tickets.md`). «Прибывания» — учёт в реальном времени: вместо ручного ввода итога оператор стартует и стопает каждый пуск отдельно, система сама считает количество и сумму.

Тариф — обычная запись **Tariff**, та же сущность и **те же правила и лимит**, что у `counters`/`launches` — до 2 тарифов на зону, создаются/редактируются через карточку «Тарифы» и её «Добавить тариф» (запрос пользователя 2026-07-17: «здесь действуют те правила и лимит тарифов»; до этого тариф был жёстко привязан к активу через `Tariff.assetId` — пересмотрено). У зоны «Прибывания» тариф несёт два дополнительных вида, выбирает владелец при создании/редактировании:

- **За вход** (было «fixed») — фиксированная цена за пуск.
- **По факту** (было «per_minute») — цена по фактически занятому времени.

Активы и тарифы создаются **независимо друг от друга**: актив можно завести раньше тарифа (или наоборот), а привязка — отдельное действие владельца (необязательное поле «Тариф» в кебабе актива, запрос пользователя 2026-07-17: «можно создавать и то и то независимо... владелец потом сам выберет тариф к активу»). Несколько активов МОГУТ ссылаться на один и тот же тариф зоны. Актив без выбранного тарифа не может начать пуск — только это ограничивает старт, а не отсутствие тарифов в принципе.

В интерфейсе выбор режима учёта — единый список из пяти («Счётчики»/«Пуски»/«Только касса»/«Прибывания»/«Билеты», `docs/spec/10-tickets.md`), без второго уровня.

Переключение `accountingMode` свободное, действует с момента переключения: прошлые сдачи итогов неизменны. Редактируется только пока у зоны нет ни одной сдачи итогов (см. 01-counters.md, «Режим учёта зоны»).

## Тариф (один из двух видов, выбирает владелец)

- **ЗА ВХОД** — НЕСКОЛЬКО вариантов «длительность + фиксированная цена» на один тариф, не одна пара (запрос пользователя 2026-07-17: «время в игровой стоит 10 руб. за 1 час»→уточнение того же дня: «за вход он выбирает сколько его ребёнок будет: 1 час, 2 часа...» — оператор выбирает конкретный вариант при старте пуска). Владелец может добавлять/удалять/редактировать варианты в форме тарифа, минимум один вариант обязателен. У каждого варианта — своя длительность в минутах (обязательное поле) и своя цена; у активного пуска обратный отсчёт от длительности выбранного варианта, по истечении — сигнал оператору (звук + вибрация + тайл краснеет и пульсирует). Пуск НЕ завершается автоматически по истечении — только оператором. Хранятся как отдельная сущность `TariffOption` (durationMinutes, price, order), привязанная к `Tariff` (`onDelete: Cascade`) — не поля самого `Tariff`.
- **ПО ФАКТУ** — тариф за минуту фактического пребывания (запрос пользователя 2026-07-17: «время в игровой стоит 1 рубль за 1 минуту»), одна цена на тариф (без вариантов), стоимость = длительность × тариф. Округление длительности до минуты — всегда вверх (в пользу точки, без выбора владельца: округление вниз/математически недодаёт выручку за фактически занятое время). Минимальной суммы пуска нет (была обязательным полем, убрана тем же днём — «вообще не нужна, это лишнее»): `Tariff.minAmount`/`Launch.minAmountSnapshot` остаются в схеме nullable ради старых записей, у новых тарифов «По факту» всегда `null`. Живой тайл показывает время и текущую сумму справочно.

Простое редактирование на месте, **без истории цен** — как у обычного тарифа `counters`/`launches`, и то же верно для вариантов «За вход»: стоимость уже идущего пуска всё равно фиксируется снапшотом на `Launch` в момент старта (`priceSnapshot`/`durationMinutesSnapshot` — с выбранного варианта, не с самого тарифа), не читается заново, поэтому смена/удаление вариантов не задевает уже запущенные пуски. Сохранение списка вариантов при редактировании — полная замена набора (удалить прежние, создать заново по присланному списку), не точечный diff по id.

`Tariff.price` у «За вход» не используется (хранится 0 как незначащий placeholder — поле NOT NULL нужно другим режимам) — вся цена в `TariffOption`. Внутренние значения `Tariff.pricingMode` в БД не переименованы («fixed»/«per_minute» — та же математика расчёта, поменялись только подписи владельцу); `null` у обычных тарифов `counters`/`launches`.

Момент оплаты система НЕ фиксирует (не POS, деньги принимаются оператором на месте вне системы) — считается только расчётная выручка, контроль — по разнице при сдаче итогов, как в `counters`/`launches`.

## Пуск

Привязка: к активу зоны — обязательна; у актива должен быть выбран тариф зоны (см. выше), иначе взять цену неоткуда и старт невозможен. Даже одиночная игровая комната заводится как один актив зоны (например, «ХалаБуда») — не как зона без активов.

Один актив может держать несколько параллельных открытых пусков (несколько детей одновременно) — физически это браслеты с номерами, которые оператор выдаёт на входе (запрос пользователя 2026-07-17). Номер (`Launch.number`) — **наименьший свободный СРЕДИ ОТКРЫТЫХ пусков этого актива**, не бесконечно растущий счётчик: если сейчас заняты 1/2/3 и освобождается 2 (пуск завершён), следующему присваивается 2, а не 4 (`src/lib/game-room.ts`, `smallestFreeNumber`). Присваивается **сервером атомарно** через advisory-lock по `assetId` (два одновременных старта с разных устройств не должны получить один номер). Пул номеров **отдельный на каждый актив** зоны — один и тот же номер может быть одновременно активен на двух разных активах, это не конфликт (запрос пользователя того же дня: «отдельный пул браслетов на каждый актив»). Свободного текстового поля-метки при старте больше нет — номер и есть идентификатор браслета, отдельно ничего не подписывается.

Пауз нет: время течёт от старта до стопа. Источник времени — **сервер**, тот же паттерн, что открытые смены авто-режима (`docs/spec/05-work-time.md`): `startedAt`/`endedAt` ставятся сервером, клиентский таймер — только отображение от серверного `startedAt`, переживает перезапуск PWA и смену устройства.

## Жизненный цикл

Старт и стоп — только оператор, в реальном времени. Владелец НЕ следит за отдельными пусками и не правит/аннулирует их (решение пользователя 2026-07-17: «владельцу не надо следить за этим вообще») — ни экрана живого мониторинга, ни списка пусков, ни порога «долгого пуска» нет. Если оператор забыл остановить пуск — closing остаётся его же действием (задним числом, когда вспомнил); коррекция расчётной выручки происходит естественно через саму механику стоп-события, без отдельного владельческого вмешательства.

Автостопа нет — время не выдумывается.

## Деньги и сдача итогов

*Согласующие правки в `docs/spec/01-counters.md`, раздел «Сдача итогов» — см. отдельный список изменений ниже.*

Пуски не создают операций в журнале Деньги поштучно — как и показания `counters`. Для зоны `accountingMode = "stays"`:

- Шаг 2 мастера сдачи итогов (показания/заезды) в исходном смысле («ввести число со счётчика») не применяется — как у `cash_only`. Но, в отличие от него, расчётная выручка и разница существуют, и с 2026-07-17 (запрос пользователя: «Активы по аналогии, как и со счётчиками») это место в UI занимает СЕТКА ТАЙЛОВ АКТИВОВ — визуально тот же паттерн, что у `counters`/`launches`, просто внутри не показание, а касса конкретного актива (см. ниже).
- Расчётная выручка = агрегат завершённых пусков **с момента предыдущей сдачи, по каждому активу отдельно**:
  - **за вход**: количество завершённых пусков × цена;
  - **по факту**: сумма стоимостей завершённых пусков (у каждого — уже применённые округление и минималка на момент закрытия).
- Шаг 3 мастера («возвраты/тестовые») для зон `accountingMode = "stays"` не показывается — его роль выполняет аннулирование пуска владельцем (аннулированный пуск целиком исключается из агрегата). У `counters`/`launches` возвраты/тестовые остаются, без изменений.
- **Шаг 4 (касса) — по каждому активу отдельно**, не одной парой полей на зону (запрос пользователя 2026-07-17: «внутри актива... сумма расчётная как read only, сотрудник вносит реальные суммы — так мы узнаем есть ли разница»). Тайл актива → sheet: расчётная сумма актива READ-ONLY сверху, ниже — поля Наличные/Безнал, куда сотрудник вносит реально собранное; под ними — живая «Разница» (факт минус расчёт), синим цветом при совпадении, предупреждающим — при расхождении, не блокирует сохранение. Тайл получает зелёную галочку, как только оба поля заполнены (тот же визуальный язык, что у заполненного показания счётчика). Сумма всех активов зоны становится общей `cashAmount`/`mobileAmount` зоны — отправка на сервер (`ZoneSubmission`) не меняется, это только способ ввода. Шаг 5 (расходы) — без изменений.
- Разница = фактическая касса (наличные + безнал) − расчётная выручка; вся логика контроля (кто видит, что делает владелец) — как в `counters`/`launches`, теперь видна ещё и по каждому активу отдельно в момент ввода, не только по зоне в отчёте.

**Мягкая блокировка**: сдача итогов по зоне недоступна при открытых пусках — сообщение «Заверши N активных пусков», без обхода.

В отчётах пуски и показания `counters`/`launches` агрегируются как единые «сеансы»; для зон `accountingMode = "stays"` дополнительно доступно суммарное время (Xч Yм).

Сводка (Telegram/email) после сдачи итогов — для зон `accountingMode = "stays"` вместо блока показаний: «Пусков: N · время: Xч Yм».

## Экран зоны в PWA

Полностью переработан 2026-07-17 (запрос пользователя: «браслеты — тайлы, а не спрятаны за тапом по активу») — заменяет прежнюю раскладку «сетка активов → sheet со списком пусков».

**Вход.** Пункт «Прибывания» в нижнем баре оператора (`docs/spec/03-design-system.md`, «Навигация») — рендерится, только если у оператора есть доступ хоть к одной зоне режима `stays` (выключенный модуль не рендерится). Одна такая зона — сразу открывается её экран, без промежуточного списка; несколько — сначала список зон на выбор (`/operator/game-room`), тот же принцип, что был раньше карточкой на главном экране оператора (убрана оттуда тем же запросом — вход теперь только через бар).

**Экран зоны.** Наверху — dropdown-переключатель активов зоны (если актив один — просто статичная подпись, без интерактивного переключателя). Ниже — сетка тайлов **браслетов ТОЛЬКО выбранного в переключателе актива** (не всех активов сразу): один тайл на каждый открытый пуск (номер, обратный отсчёт/счётчик времени вперёд по тарифу «За вход»/«По факту», текущая сумма справочно для «По факту») + один тайл «Добавить браслет» в конце сетки.

Тайл «Добавить браслет»: тап → старт нового пуска на выбранном активе.

- **«По факту»** — старт мгновенный (старт не должен требовать более двух тапов), способ оплаты спрашивается позже, при остановке (сумма известна только тогда, см. ниже).
- **«За вход»** — цена известна заранее, поэтому оплата берётся сразу при старте (запрос пользователя 2026-07-17: «надо сразу при старте браслета спрашивать Наличными или Безнал»), не мгновенно: тап открывает sheet — если у тарифа несколько вариантов длительности, сначала их выбор, затем (в том же sheet, один поток, запрос того же дня: «в одном») выбор способа оплаты (Наличные/Безнал); если вариант один — sheet сразу открывается на шаге оплаты. Пуск создаётся с уже заполненным `Launch.paymentMethod`.

Актив без выбранного тарифа — тайл «Добавить браслет» неактивен, над сеткой видна причина.

Тайл открытого браслета: тап → **сразу**, ВНУТРИ ТОГО ЖЕ ТАЙЛА (не в отдельном sheet, запрос пользователя 2026-07-17: «вопрос "Точно" должен появляться внутри тайла») содержимое сменяется на подтверждение — сумма («по факту») и вопрос «Точно?» с кнопками ✕ (отмена, назад к обычному виду тайла) / ✓ (подтвердить). Второй тап обязателен, случайный тап по тайлу не останавливает пуск сразу.

- **«За вход»**: подтверждение сразу останавливает пуск — способ оплаты уже зафиксирован при старте, повторно не спрашивается.
- **«По факту»**: подтверждение переводит тот же тайл в третье состояние — выбор способа оплаты (Наличные/Безнал, те же подписи, что на шаге «Касса» мастера сдачи итогов), тап по одному из двух сразу останавливает пуск с этим способом (запрос пользователя того же дня: «это только касается тарифа По факту»).

Способ оплаты (`Launch.paymentMethod`, "cash"/"mobile") — у «За вход» ставится при СТАРТЕ (см. выше), у «По факту» — при ОСТАНОВКЕ; null только пока пуск «По факту» ещё открыт. **Только справочная разбивка**, не автозаполнение. `aggregateGameRoomLaunches` суммирует её в `cashAmount`/`mobileAmount` (совпадает с `totalAmount`, когда способ оплаты указан у всех завершённых пусков окна). На шаге «Касса» мастера сдачи итогов оператору показывается подсказка с этой разбивкой рядом с полями Наличные/Безнал, но сами поля остаются ручным вводом, как и были — автоподстановка стёрла бы контроль недостачи через «Разницу» (решение пользователя 2026-07-17: физический пересчёт должен сверяться с расчётом независимо, а не сверять систему саму с собой).

Истёкший отсчёт («за вход» с длительностью): звук + вибрация + пульсация тайла браслета, независимо от того, какой актив сейчас выбран в переключателе (сигнал по всей зоне, не только по видимому активу). Звук воспроизводится только при открытом PWA (Web Audio, разблокируется первым взаимодействием пользователя с приложением) — ограничение платформы, зафиксировано здесь и в подсказке оператору при первом использовании режима. `prefers-reduced-motion` — без пульсации, сигнал остаётся звук + вибрация.

Вёрстка по `docs/spec/03-design-system.md`, все строки — через `/lang`.

## Кабинет владельца

1. **Настройка зоны**: режим учёта (единый сегмент из пяти: «Счётчики»/«Пуски»/«Только касса»/«Прибывания»/«Билеты») и тарифы — карточка «Тарифы», та же, что у Счётчиков/Пусков (до 2 на зону; кнопка «Добавить тариф» скрывается целиком при достижении лимита, не показывается disabled — запрос пользователя 2026-07-17), с доп. видом За вход (список вариантов длительность+цена, минимум один, можно добавлять/удалять/редактировать) / По факту (одна цена/минуту, без минималки).
2. **Настройка актива**: выбор тарифа зоны (необязательное поле, можно оставить пустым и привязать позже) — в кебабе актива, вместе с обычными полями (имя, цвет, фото).

Владелец НЕ видит отдельные пуски — ни живого статуса «сейчас на точке», ни списка пусков за период (решение пользователя 2026-07-17). Наблюдение владельца за режимом «Прибывания» ограничено агрегатом при сдаче итогов (расчётная выручка, разница) — как у любого другого режима учёта.

## Ограничения MVP (осознанные решения)

- Офлайна нет — без сети старт/стоп невозможны.
- Telegram/email-уведомлений по каждому отдельному пуску нет — только агрегат в сводке сдачи итогов.
- Паузы нет.
- Тариф актива один, выбирается владельцем заранее (актив ссылается ровно на одну запись `Tariff`, хотя несколько активов могут разделять один и тот же тариф) — оператор при старте выбирает только вариант «За вход» ВНУТРИ этого тарифа (длительность+цена), не сам тариф.

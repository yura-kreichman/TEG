# Архитектурные принципы

Сквозной раздел — применяется ко всем модулям, а не только к какому-то одному.

## Модульность

Система строится как ядро + функциональные области («модули» — термин кодовой организации, не единица продажи). Система универсальна: **у каждого тенанта потенциально доступна вся функциональность целиком** — пакет подписки/лимиты НИКОГДА не решают, какие области видны, разница пакетов исключительно в числовых лимитах (см. «Подписки и лимиты» ниже). Это не значит, что тумблеров нет вообще: для части областей (Инструктажи, Задачи, Лендинг, Товары, Клиенты/Абонементы — `Tenant.*Enabled`, `src/lib/tenant-modules.ts`) Владелец САМ может включить/выключить область для своего тенанта из Настроек → Система, если она ему не нужна — это его собственное решение об интерфейсе, а не оверрайд со стороны Super Admin/пакета и не привязка к тарифу (выключенный на Starter модуль точно так же можно выключить на Max).

**Ядро:** тенанты, пользователи и роли (Super Admin, Owner, Operator), точки, зоны, активы, настройки.

**Функциональные области** (список открытый): Счётчики, Деньги, Рабочее время, Задания, Прибывания, Абонементы, Товары, Билеты, Отчёты.

Добавление новой области не должно требовать изменения ядра. Каждая область объявляет:
- свои сущности/таблицы;
- свои экраны — в PWA оператора и в кабинете владельца;
- права ролей;
- свои типы записей в общие журналы (например, денежные операции).

Это разделение — исключительно про структуру кода (что где живёт, как расширяется без правок ядра), не про пакет/подписку: включать/выключать области ПО ПАКЕТУ нельзя, такого переключателя на уровне подписки в системе нет и не будет — см. владельческий тумблер отдельных модулей выше, он не имеет отношения к пакету/лимитам.

## Навигация

НИЖНИЙ БАР ВЛАДЕЛЬЦА. Состав: Главная · Деньги · Отчёты · Операторы · Ещё. Бар собирается динамически из модулей, включённых у тенанта (владельческий тумблер Настроек → Система, см. «Модульность» выше — НЕ пакет/подписка, у пакета такого переключателя нет). Приоритетный порядок пунктов: Главная, Деньги, Отчёты, Операторы, Задачи, Точки — первые четыре доступных занимают слоты бара, пятый слот всегда «Ещё». Если модуль пункта выключен владельцем (например, Задачи), его слот занимает следующий по приоритету пункт — выключенные пункты не показываются. В «Ещё» (bottom sheet): Точки, Задачи, Настройки и все остальные включённые модули; Настройки — всегда в «Ещё». На «Ещё» отображается badge-точка, если внутри есть пункты с активными событиями (например, невыполненные задачи). Пункт «Точки» из бара переносится в «Ещё»: это конфигурация, а не ежедневный сценарий.

## Подписки и лимиты

Пакет подписки = **только** числовые лимиты: количество точек, зон, активов, операторов (см. пример пакета «Max» — карточка «Ваш план» в кабинете владельца). Никакого набора «включённых модулей» пакет не содержит — вся функциональность доступна на любом пакете, разница исключительно в лимитах. Лимиты проверяются при создании сущностей (например, «в вашем пакете до 15 счётчиков»).

Статусы подписки: `active`, `paused` (сезонная пауза), `expired`.

Конфигурация пакетов не зашита в код — управляется из админ-модуля.

## Админ-модуль (Super Admin)

Отдельный интерфейс владельца платформы:
- тенанты и их статусы;
- конфигуратор пакетов (лимиты + цены — без выбора модулей, см. «Подписки и лимиты»);
- ручные оверрайды лимитов конкретному тенанту (индивидуальные условия);
- базовая статистика использования.

## Мультиязычность

Все строки UI — в JSON-файлах в `/lang` (`ru.json`, `en.json`, `ro.json`...). В коде — только ключи.

Язык — настройка тенанта с возможностью личного переопределения пользователем.

Пользовательские данные (названия зон, активов, тарифов, комментарии) не переводятся.

Базовый язык разработки — **ru**.

## Темы

Светлая и тёмная тема во всех интерфейсах, переключение пользователем, реализация через CSS-переменные (design tokens) — модули наследуют тему автоматически.

## Мультитенантность входа

PIN оператора уникален только внутри тенанта; у разных тенантов PIN могут совпадать.

Аккаунты операторов создаёт владелец.

Привязка сессии/устройства к тенанту — один из двух способов:
- установочная ссылка/QR точки (PWA ставится по ней один раз, например при найме оператора);
- активированное владельцем «устройство точки».

Деактивация PIN мгновенно закрывает доступ.

## Терминология

«Сдача итогов» (results submission) — кассовое событие модуля Счётчики.

Слово «смена/shift» зарезервировано за будущим модулем «Рабочее время» и в модуле Счётчики **не используется**.

## Аналитика и внешние сервисы

Статистика и аналитика — только собственные, внутри платформы. Внешние аналитические сервисы, счётчики и сторонние скрипты на страницах запрещены. Гео-данные — по локальной базе (GeoLite2-файл) без обращений к внешним API и без хранения IP.

Внешние API допустимы только server-side как инструменты владельца; на публичных страницах — ноль внешних запросов.

Встраивание внешнего контента допустимо только по схеме фасада: до явного клика посетителя — ноль внешних запросов со страницы.
